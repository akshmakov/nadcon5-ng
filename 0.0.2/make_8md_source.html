10 # NADCON5.0 Data Pipeline {#sdatapipe}
11 
12 
13 The toplevel \ref Makefile defines the rules to construct the NADCON5-ng dataset in the traditional NADCON5.0 fashion, using the same fortran programs, simply managed by Make
14 
15 To generate the target NADCON data, from the command line
16 
17  $ cd nadcon5-ng
18  $ make OLD_DATUM=ussd NEW_DATUM=nad27 REGION=conus GRIDSPACING=900 MAPLEVEL=0
19  $ cd build/out.ussd.nad27.conus.900.0
20 
21 Where the variables are specified and indicate
22 
23  - `OLD_DATUM` = Source Datum
24  - `NEW_DATUM` = Target Datum
25  - `REGION` = Geographic Region to Compute
26  - `GRIDSPACING` = The Grid Spacing in arcsec
27  - `MAPLEVEL` = The zoom level of images to generate (`0,1,2`)
28 
29 For more Information on the meaning of thes parameter see the documentation of functions in: \ref doers
30 
31 \note Conversion between datums is restricted in a strictly one-step chronoligical direction. Not all Regions are possible with all conversions
32

34 ## Step-by-step {#ss-step-by-step}
35 
36 With no arguments Make will execute the default target, in this case `all`, which is defined to run the data pipeline.
37 
38 Additional targets are defined to step through the data pipeline.
39 
40 These targets approximately mimic the steps taken in the upstream build scritpts `doitX.bat`
41 
42  $ make doit
43  $ make doit2
44  $ make doit3
45  $ make doit4
46 
47 
48 , will execute the build system in the same step-by step fashion as the upstream batch files. Generating a portion of the output each time.
49 
50 These `doit` targets use programs defined in \ref doers to generated Generic Mapping Tools (GMT) batch files in the output directory. Additionally, as a first step a "work" file is constructed (see \ref makework )
51 
52 Dataset construction can be done by stepping through these GMT scripts
53 
54  $ make workfile
55  $ make gmtbat01
56  $ make gmtbat02
57  ...
58  $ make gmtbat07
59 
60 Allowing one to manually execute the batch file in the build directory
61 
62 \note These scripts call GMT functions without explicitly calling `gmt` which may not work on all distributions, shell wrappers for gmt are provided in `gmt_wrappers/` and can be added to path for convenience.

145 # Makefile Primer {#s-make-primer}
146 
147 **Note:** This primer has been adapted from other sources and is not specific to `NADCON5-ng`
148 
149 Makefiles define **rules** to make **targets**.
150 
151 A Makefile may look something like this.
152 
153 
154 
155  # Target to Compile a single file.
156  # The first line defines the target
157  # further lines define the commands
158  # that are run
159  object.o:
160  cc -o object.o object.c
161 
162  # Target to link executable with external releaselib.
163  # object.o is a dependency of release_exec
164  release_exec: object.o
165  ld -r -o release_exec releaselib object.o
166 
167  # Convenience Target
168  release: release_exec
169 
170  # Since "release" does not actually produce a file
171  # this is required boiler plate
172  .PHONY: release
173 
174 
175 This Makefile is actually equivalent to a simple compiler one-liner.
176 
177 
178  cc -o release_exec -lreleaselib object.c
179 
180 
181 However, it already provides power behind the scenes.
182 
183 For example Calling
184 
185 
186  make release
187 
188 
189 Will not recompile the executable if `object.c` has not been changed.
190 
191 The real power of `Make` comes from "Pattern Rules" constructed using [Automatic variables](https://www.gnu.org/software/make/manual/html_node/Automatic-Variables.html)
192 and [Pattern Matching](https://www.gnu.org/software/make/manual/html_node/Pattern-Match.html#Pattern-Match) which result in a more general rule.
193 
194 
195  # The target name is special
196  # and the variable $@
197  # refers to the target name
198  test.o:
199  cc -o $@ object.c
200 
201  # The dependency name is special
202  # and the variable $< refers to the
203  # first (only) dependency
204  better_test.o: object.c
205  cc -o $@ $<
206 
207  # Multiple Dependencies
208  # can be refered to by the variable
209  # $^
210  other_test.a: test.o better_test.o
211  ld -r -o $@ $^
212 
213  # Pattern matching makes general rules
214  # The following rule compiles all .c files to .o files
215  %.o:%.c
216  cc -o $@ $<
217 
218  # PHONY rules can be used to build multiple
219  # dependencies
220  build_some_libs: pattern_lib.o \
221  pattern_lib2.o \
222  pattern_lib3.o \
223  other_test.a
224  .PHONY: build_all_libs
225 
226 
227 
228 By default `Make` assumes that the target is a real file that is created, and will track the state of
229 dependency based on the file and if it exists.
230 
231 As such when dealing with build directories, a target must be specified with its path for make to track its status.
232 
233 The `.PHONY` target is a special target that indicates that
234 it should not be tracked and will be rebuilt every time, often this is good for things like printing debug output
235 or a convenience target that builds multiple targets (e.g. `all`).
236 
237 
238 In our Makefiles you will see things like the following, this example is our default rule for building fortran programs
239 
240  $(BIN_DIR)/%:$(notdir %).f | $(BIN_DIR)
241  $(FC) $(FFLAGS) $< -o $@
242 
243 
244 
245 In this case, `FC`, `FFLAGS`, are variables define in our Makefile which can be overridden at the command line, this is described below. `BIN_DIR` is a Makefile Variable and is the directory where binary programs artifacts are to be placed after building.
246 
247 This default rule maps all programs like `/path/to/build/bin/myprogram` to compile from the file `myprogram.f` using the Fortran Compile `FC` and the compilation flags `FFLAGS`
